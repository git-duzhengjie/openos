# A2.P2 — wait / waitpid 设计文档

## 目标
让 ring3 用户程序可以：
```c
int pid = fork();
if (pid == 0) { /* child */ exit(7); }
int status;
int waited = wait(&status);   // 期望 waited == 2, WEXITSTATUS(status) == 7
```

当前 fork 模型（A2.P3-B 落地的）是 **vfork + 强制 exit**：
- parent 进入 SYS_FORK → 在 PCB 中置位 `fork_pending=1`，记下 user rsp
- syscall 直接返回 `child_pid=2` 给 parent
- parent 继续跑 ring3 直到 SYS_EXIT；exit 路径检测到 `fork_pending` 后不释放 PCB，转而让主循环重新 `usermode_run`，走 fork-resume 路径
- fork-resume 用同一个 PCB、同一个用户栈，但把 rax 改为 0 让 child 路径执行
- child 最终 SYS_EXIT 真正退出

→ 因此当前 parent **必须先 SYS_EXIT 才能让 child 跑**，wait() 在此模型下不可用。

## 设计方案

走"**parent 在 SYS_WAIT 中嵌套驱动 child**"的简化路径，避免引入完整 scheduler：

### PCB 字段增量
```c
struct arch_x86_64_proc {
    /* ... 既有字段 ... */

    /* A2.P2 — wait/waitpid */
    uint32_t child_pid;          /* 由 fork 写入；wait 读取 */
    uint8_t  child_exited;       /* child 调用 SYS_EXIT 后置 1 */
    int32_t  child_exit_code;    /* child 给 SYS_EXIT 的 status */
    uint8_t  wait_in_progress;   /* parent 正在 do_wait 内嵌驱动 child */
};
```

### fork 改造
保留 vfork 语义，但额外：
1. `fork_pending=1` 时把 `p->child_pid = next_pid()` 记下来（不再硬编码 2）
2. 用户态返回值 = `child_pid`

### SYS_EXIT 改造（关键）
`do_exit` 现在的逻辑：检测 `fork_pending` 决定是否真退出。新逻辑：

```
if (wait_in_progress) {
    /* 我们是 child（嵌套上下文），写父PCB字段后 longjmp 回 do_wait */
    parent_pcb->child_exited = 1;
    parent_pcb->child_exit_code = status;
    /* 走 fork-resume 已用过的 mark_exited+return_to_kernel 路径 */
    /* 但目标 anchor 是 do_wait 的 setjmp 点，不是主循环 */
    longjmp(parent_pcb->wait_anchor, 1);
}
else if (fork_pending) {
    /* 旧的 vfork 强制 resume 路径（保留向下兼容，未来废弃） */
    ...
}
else {
    /* 真正退出 */
    ...
}
```

### SYS_WAIT 实装
```c
static int64_t do_wait(uint64_t status_user_ptr) {
    proc_t *p = current_proc();
    if (!p->fork_pending && !p->child_exited) return -1;  /* no child */

    if (p->child_exited) goto reap;

    /* parent 进入嵌套驱动 child 模式 */
    p->wait_in_progress = 1;
    if (setjmp(p->wait_anchor) == 0) {
        /* 把 fork_pending 转换成"现在就跑 child" */
        usermode_run_child_from_fork(p);
        /* 不应到达 */
    }
    /* longjmp 跳回来后 */
    p->wait_in_progress = 0;
    p->fork_pending = 0;

reap:
    if (status_user_ptr) *(int*)status_user_ptr = p->child_exit_code << 8; /* W_EXITCODE */
    int32_t pid = p->child_pid;
    p->child_pid = 0;
    p->child_exited = 0;
    return pid;
}
```

### usermode_run_child_from_fork()
新增内核函数，复用 fork-resume 已有的逻辑：
- 用 PCB 中 `fork_user_rsp` 和 `resume_frame` 准备 child 的 ring3 上下文
- 把 rax 置 0
- 调 `arch_x86_64_usermode_run(child_entry)`，但 entry 是 fork-resume 模式
- 如果 child 没 longjmp 而是正常返回，记为异常并设 child_exit_code=-1

### setjmp / longjmp 内核侧
若内核当前未有 setjmp/longjmp，可使用：
- 已有 `arch_x86_64_usermode_return_to_kernel()`（基于 PCB.kernel_return_rsp）
- 增加一个 `arch_x86_64_usermode_return_to_wait_anchor()`，目标 RSP 是 do_wait 局部保存的栈位置

更简单方案：do_wait 不嵌套，而是把 wait_pending 置位后 longjmp 回 main_loop，main_loop 重新 usermode_run(child)，child SYS_EXIT 后再 usermode_run(parent) resume。但这种 round-trip 模型需要 parent ring3 trapframe 完整保存——当前 syscall 路径已经保存（在 PCB.saved_user_frame），可行。

## 实装顺序

1. **P2.a** — PCB 增字段 + fork 写 child_pid（不改语义）
2. **P2.b** — do_wait 框架（仅处理 child_exited 已置位的情况，即"延迟收割"）
3. **P2.c** — child 驱动机制（嵌套或 round-trip 二选一）
4. **P2.d** — hello64_v3 用户测试程序（fork + wait + 校验 status）

### P2.c 路径再决策
- **嵌套路径**：do_wait 内直接调 usermode_run，child exit 时 longjmp 跳回 do_wait setjmp 点
  - 优点：parent ring3 上下文不用额外保存（栈在内核 do_wait 帧里）
  - 缺点：需要内核 setjmp / 自定义 anchor
- **Round-trip 路径**：do_wait 设 wait_pending，longjmp 回主循环；主循环调度 child→parent
  - 优点：复用已有 fork-resume 基础设施
  - 缺点：需要 parent ring3 trapframe + 主循环新增状态机

倾向**嵌套路径**：与 fork-resume 已有 mark_exited 机制最一致，只需新增一个 anchor。

## 验收标准
- hello64_v3：fork → child exit(7) → parent wait → 打印 `[parent] child pid=2 status=7`
- QEMU 不 reboot、cpl=3 deref ok、`#PF` count=0
- 既有 hello64_v2 fork 测试仍 pass（vfork 路径未废弃）
